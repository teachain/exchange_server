## 1. 项目概述

### 1.1 项目背景

ViaBtc Exchange Server 是一个高性能的加密货币交易所后端系统，采用微服务架构设计，支持订单匹配、交易执行、市场数据发布、实时WebSocket推送等功能。

### 1.2 系统目标

- **高性能**: 支持高并发的订单处理和交易匹配
- **可扩展**: 微服务架构，便于横向扩展
- **高可用**: 支持服务状态恢复，Kafka消息队列解耦
- **实时性**: WebSocket推送实时市场数据和订单更新

### 1.3 技术栈

| 组件 | 技术选型 |
|------|----------|
| 编程语言 | Go 1.24.0 |
| 数据库 | MySQL (分表100张) |
| 消息队列 | Apache Kafka |
| 缓存 | Redis |
| HTTP框架 | Gin |
| 序列化 | Protocol Buffers, JSON |

## 3. 数据库设计

### 3.1 trade_log 数据库

**用途**: 运行数据存储，包括订单快照、余额快照、操作日志

| 表名 | 分片方式 | 说明 |
|------|---------|------|
| `operlog_YYYYMMDD` | 按日期分区 | 操作日志表 (883张) |
| `slice_balance` | - | 余额快照主表 |
| `slice_balance_*` | - | 分时余额快照 |
| `slice_order` | - | 订单快照表 |
| `slice_history` | - | 分片进度记录 |

### 3.2 trade_history 数据库

**用途**: 历史数据存储，100表分片

| 表名 | 分片键 | 说明 |
|------|--------|------|
| `balance_history_0~99` | user_id % 100 | 余额变更历史 |
| `order_history_0~99` | user_id % 100 | 订单完成历史 |
| `order_detail_0~99` | order_id % 100 | 订单详情 |
| `deal_history_0~99` | order_id % 100 | 成交历史 |
| `user_deal_history_0~99` | user_id % 100 | 用户成交历史 |

## 4. Kafka消息队列

### 4.1 Topic定义

| Topic | 生产者 | 消费者 | 用途 |
|-------|--------|--------|------|
| `orders` | MatchEngine | AccessWS | 订单事件推送 |
| `deals` | MatchEngine | MarketPrice, AccessWS | 成交事件推送 |
| `balances` | MatchEngine | AccessWS | 余额变更推送 |

### 4.2 消息格式

**Orders Topic**:
```json
{
  "event": 1,
  "order": {
    "id": 123,
    "user_id": 1,
    "market": "BTC_USDT",
    "side": 1,
    "type": 1,
    "price": "50000.00",
    "amount": "1.5",
    "left": "0.5"
  }
}
```

**Deals Topic**:
```json
{
  "trade": {
    "id": 456,
    "market": "BTC_USDT",
    "price": "50000.00",
    "amount": "1.0",
    "side": 1,
    "created_at": 1709510400
  }
}
```

## 5. 服务间通信

### 5.1 HTTP/REST
- AccessHTTP 作为API网关，通过HTTP JSON-RPC转发请求到后端服务

### 5.2 自定义RPC协议
- AccessWS 使用自定义二进制RPC协议查询后端
- TCP连接，自定义编解码 (8字节头 + body)

### 5.3 Kafka
- 异步事件流，用于实时推送
- AccessWS 订阅 orders、balances 主题

## 7. 数据流

### 7.1 订单处理流程

```
Client → AccessHTTP → MatchEngine → Kafka (orders/deals/balances)
                              ↓
                         trade_log
                              ↓
                       MatchEngine response
                              ↓
                         AccessHTTP → Client
```

### 7.2 实时推送流程

```
MatchEngine → Kafka → AccessWS → WebSocket → Client
                      ↓
                  MarketPrice → AccessWS → WebSocket → Client
```

### 7.3 历史查询流程

```
Client → AccessHTTP → ReadHistory → trade_history → Client
```

## 8. 系统恢复机制

### 8.1 状态恢复

MatchEngine 支持通过以下机制恢复状态:

1. **Slice文件**: 定期将订单/余额快照写入磁盘
2. **OperLog**: 操作日志重放
3. **MySQL表**: slice_order, slice_balance, operlog

### 8.2 恢复流程

1. 加载最新的Slice文件
2. 恢复订单和余额
3. 从 end_oper_id 开始重放 operlog
4. 恢复正常运行
